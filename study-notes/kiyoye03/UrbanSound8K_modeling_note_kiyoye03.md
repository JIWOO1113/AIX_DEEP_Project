# UrbanSound8K 모델링 활용 방향 정리

## 1. 데이터셋 이해

UrbanSound8K는 도시 환경에서 발생하는 다양한 소리를 분류하기 위한 오디오 데이터셋이다.  
각 데이터는 wav 형식의 짧은 오디오 파일이며, 하나의 파일은 10개의 도시 소리 클래스 중 하나로 라벨링되어 있다.

이 데이터셋은 이미지나 텍스트 데이터와 달리, 오디오 파일 자체를 바로 모델에 넣기보다는 먼저 소리의 특징을 추출한 뒤 모델 학습에 사용하는 방식이 필요할 것 같다.

따라서 이 프로젝트는 기본적으로 도시 소리를 분류하는 multi-class classification 문제로 볼 수 있다.

---

## 2. 입력값과 예측 대상

이 프로젝트에서 입력값은 기본적으로 오디오 파일이다.  
하지만 실제 모델 학습에서는 wav 파일에서 추출한 특징값을 입력으로 사용할 수 있다.

| 구분 | 내용 |
|---|---|
| 원본 입력 데이터 | wav 오디오 파일 |
| 변환 후 입력 데이터 | MFCC, Mel-spectrogram 등 오디오 특징 |
| 예측 대상 | classID 또는 class |
| 문제 유형 | multi-class classification |

즉, 이 데이터셋은 특정 수치를 예측하는 regression 문제가 아니라, 여러 개의 소리 종류 중 하나를 예측하는 classification 문제이다.

---

## 3. metadata의 역할

UrbanSound8K에서 metadata 파일은 오디오 파일과 라벨을 연결해주는 기준표 역할을 한다.  
실제 wav 파일만 보면 어떤 소리인지 바로 알 수 없기 때문에, metadata를 통해 파일명과 class 정보를 함께 확인해야 한다.

중요하게 볼 수 있는 컬럼은 다음과 같다.

| 컬럼 | 활용 이유 |
|---|---|
| slice_file_name | 실제 오디오 파일과 metadata를 연결할 때 필요 |
| fold | train/test 분리 또는 cross validation에 활용 가능 |
| classID | 모델 학습 시 label 값으로 사용 가능 |
| class | 결과 해석 시 사람이 이해하기 쉬운 클래스 이름 |
| start, end | 원본 오디오에서 잘라낸 구간 확인 가능 |
| salience | foreground/background 여부를 참고할 수 있음 |

따라서 metadata는 단순한 설명 파일이 아니라, 모델 학습을 위해 오디오 파일과 정답 라벨을 연결하는 핵심 파일이라고 볼 수 있다.

---

## 4. fold 구조를 활용해야 하는 이유

UrbanSound8K는 fold1부터 fold10까지 미리 데이터가 나누어져 있다.  
이 구조는 단순한 폴더 구분이 아니라 모델 평가를 위해 활용할 수 있는 구조로 보인다.

전체 데이터를 무작위로 섞으면 같은 원본 오디오에서 잘라낸 비슷한 소리 조각이 train data와 test data에 동시에 들어갈 수 있다.  
그 경우 모델 성능이 실제보다 좋게 나올 가능성이 있다.

그래서 나중에 모델을 평가할 때는 기존 fold 구조를 활용하는 것이 적절해 보인다.  
예를 들어 9개의 fold는 train data로 사용하고, 나머지 1개의 fold는 test data로 사용하는 방식이 가능하다.

---

## 5. 오디오 데이터 전처리 방향

오디오 파일은 그대로 사용하기 어렵기 때문에, 모델에 넣기 전에 특징 추출 과정이 필요하다.

사용할 수 있을 것 같은 특징은 다음과 같다.

| 특징 | 설명 |
|---|---|
| MFCC | 음성/소리 분류에서 자주 사용되는 대표적인 오디오 특징 |
| Mel-spectrogram | 소리를 이미지처럼 변환해서 CNN에 사용할 수 있음 |
| Chroma feature | 음의 높낮이나 음악적 특징을 볼 때 사용 가능 |
| Spectral contrast | 주파수 대역별 차이를 나타내는 특징 |

처음에는 MFCC를 기준으로 간단한 모델을 적용하고, 이후 Mel-spectrogram을 사용해 CNN 모델과 비교하는 방식도 가능할 것 같다.

---

## 6. 모델 비교 방향

교수님이 Option A에서 여러 모델을 비교하는 방향을 말씀하셨기 때문에, 한 가지 모델만 사용하는 것보다는 2~3개 모델을 비교하는 것이 좋을 것 같다.

가능한 모델 후보는 다음과 같다.

| 모델 | 활용 방향 |
|---|---|
| Random Forest | MFCC 같은 특징값을 사용해서 비교적 간단하게 분류 가능 |
| SVM | feature 기반 분류 문제에서 기본 비교 모델로 사용 가능 |
| XGBoost | Random Forest와 함께 성능 비교용으로 활용 가능 |
| MLP | 추출된 feature를 입력으로 사용하는 기본 신경망 모델 |
| CNN | Mel-spectrogram을 이미지처럼 사용해서 분류 가능 |

처음부터 CNN만 사용하는 것보다, feature 기반 모델과 CNN을 비교하면 모델별 차이를 설명하기 더 좋을 것 같다.

---

## 7. 결과 분석 시 볼 수 있는 지표

이 데이터셋은 multi-class classification 문제이기 때문에, 단순 accuracy뿐만 아니라 클래스별 성능도 확인하는 것이 좋을 것 같다.

확인할 수 있는 지표는 다음과 같다.

| 지표 | 의미 |
|---|---|
| Accuracy | 전체 데이터 중 맞게 예측한 비율 |
| Precision | 특정 클래스로 예측한 것 중 실제로 맞은 비율 |
| Recall | 실제 특정 클래스 중 모델이 맞게 찾은 비율 |
| F1-score | precision과 recall을 함께 고려한 지표 |
| Confusion matrix | 어떤 클래스끼리 많이 헷갈리는지 확인 가능 |

예를 들어 drilling과 jackhammer처럼 비슷한 소리는 모델이 혼동할 가능성이 있다.  
따라서 confusion matrix를 확인하면 어떤 소리 클래스에서 오분류가 많이 발생하는지 해석하는 데 도움이 될 것 같다.

## 8. 공지 요구사항 기준 정리

이번 조사에서 확인해야 할 핵심은 데이터 구조를 이해하는 것이다.

| 요구사항 | 확인한 내용 |
|---|---|
| 파일이 무엇을 의미하는지 | audio 폴더는 실제 wav 오디오 파일이고, metadata는 파일명과 라벨을 연결하는 기준표 역할을 한다. |
| 숫자가 무엇을 의미하는지 | classID는 소리 종류를 나타내는 숫자 라벨이고, fold는 학습/평가 데이터 분리를 위한 번호로 볼 수 있다. |
| 이렇게 쓰인 의도 | metadata를 통해 wav 파일과 정답 라벨을 연결하고, fold 구조를 통해 더 공정하게 모델을 평가할 수 있도록 구성된 것으로 보인다. |

다음 회의 전까지는 실제 데이터를 열어보면서 아래 내용을 추가로 확인하면 좋을 것 같다.

## 10. 정리

UrbanSound8K는 데이터 구조가 비교적 명확하고, classification 문제로 정의하기 쉬운 데이터셋이다.  
다만 오디오 데이터이기 때문에 wav 파일을 바로 사용하는 것이 아니라, 특징 추출 과정이 필요하다.

따라서 프로젝트 진행 흐름은 다음과 같이 잡을 수 있을 것 같다.

1. metadata와 audio 파일 연결 확인
2. 오디오 특징 추출
3. 2~3개 모델 적용
4. 성능 지표 비교
5. 결과 해석 및 한계점 정리
